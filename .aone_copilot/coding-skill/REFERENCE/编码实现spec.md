## 首要原则（必须遵守）
*   视觉还原：页面或组件有设计图的情况下，大模型必须读取相关设计图，严格按照设计图还原。（非多模态模型则跳过）
*   交互体验：根据用户要求或者需求分析（spec.md），设计用户友好的交互体验。
*   架构要求：领域结构划分清晰，易扩展、易维护、高性能。
*   数据mock：在没有提供真实接口的情况下按照以下要求mock数据
    * 集中管理：使用单独的文件目录管理 mock 数据，按模块划分不同模块的mock数据，便于维护。
    * 区分开发环境：设计要易于替换不同环境接口数据，区分本地、预发、线上环境。
    * mock原则：本地环境默认mock查询类接口数据，列表页mock数据=(分页大小+1)条（如分页大小10，则mock数据为11条）。

## 一、技术栈约束

*   基础技术栈：如果当前工程没有指定技术栈则使用 React + Typescript。

*   基础组件：如果当前工程没有指定基础组件则使用 Ant Design。

*   状态管理：查看当前应用使用的状态管理库，并优先使用。建

*   禁止引入其它全量 UI 组件库（如 Shadcn、Element、Arco、Naive UI 等）。

*   禁止引入其它全局状态管理库（如 Redux、MobX、Recoil、Vuex/Pinia 等）。

*   禁止引入 CSS 框架替代品（如 Bootstrap、Bulma 等）以及大规模 CSS-in-JS 体系作为主方案（如 styled-components/emotion 作为主样式方案）。

## 二、分层职责（必须遵守）

### 2.1 pages（页面层）

*   目录划分：必须按业务领域划分目录，每个领域可以包含多个功能模块，当每个模块下多个页面时需按页面维护独立文件夹。
    * 充分理解业务语义，合理拆分目录结构及命名。例如：`/pages/order/order-management/order-detail`、`/pages/order/order-management/order-list`。
    * 目录名必须保持简短，原则上不超过2个单词，禁止超过3个单词，使用 - 连接符。

*   单一职责：不同页面必须不同代码文件，禁止多页面同一个文件。

*   同页面组件管理：同一个页面由多组件组成时，必须合理拆分组件；除了组件库中可以直接复用的组件外，自定义组件个数超过3个时，每个组件对应一个代码文件。

*   只负责：路由参数、页面布局、页面级编排、调用业务模块/组件。

*   禁止：写核心业务规则/复杂流程；禁止直接写接口调用（必须走 services 统一封装）。

### 2.2 components（通用组件层）

*   只放：无业务语义的通用组件（Button、Table封装、Modal封装等）。

*   禁止：直接依赖 services；禁止写业务字段语义（如 Order/User 这种业务组件不应放这里）。

*   允许依赖：components/hooks/styles。

### 2.3 hooks（通用hooks层）

*   只放：无业务语义的通用hooks（如 useDebounce/useEvent/useToggle）。

*   禁止：直接依赖 services。

*   允许依赖：hooks。

### 2.4 services（数据层）

*   目录划分：必须按业务领域划分目录，每个领域可以包含多个功能模块。

*   只负责：request封装、API调用、错误码/异常统一处理、VO/DTO->前端模型适配。

*   禁止：在 services 输出后端 VO/DTO 给 UI；必须做适配并对外提供前端可用模型。

## 三、依赖方向（强制）

*   禁止 components -> services。

*   禁止 hooks -> services。

*   禁止 services -> pages/components/hooks。

*   禁止 pages 之间互相直接引用（如 pages/A 引 pages/B）。

*   禁止 任何层出现循环依赖。

*   导入路径规则：
    *   禁止 使用超过两层的相对路径穿透导入（如 ../../..）；必须使用别名（例如 @/）。
    *   禁止 从模块内部私有路径导入；优先使用模块对外出口文件（index）导出再引用。

## 四、接口调用规范（强制）

*   任意网络请求必须从 services 发起（统一 request、统一错误处理）。

*   pages/components/hooks 不得直接拼 URL 调 fetch/axios。

*   VO/DTO适配必须在 services 完成，UI/业务层消费的是“前端模型”而不是后端原始结构。

*   错误处理不得在各处散落 try/catch + toast，应由 services 统一错误处理/用例层收敛。

## 五、状态管理规范（强制）

*   全局状态只允许放“跨页面长期存在”的信息（如用户、权限、主题、语言）。

*   服务端数据（接口数据）不要“复制一份”进全局状态；优先使用请求缓存/查询管理方案。

## 六、代码规范（强制）

*   代码格式: 强制遵循阿里巴巴前端开发规范。

*   样式布局: 优先使用 Flexbox 或 CSS Grid；避免使用 float / table 布局。 具体规范参考`.aone_copilot/rules/coding.md`中的样式规范

*   文件与职责：
    *   单文件单职责：组件/服务/适配/工具不要混写。
    *   index 文件仅做导出聚合，禁止写业务逻辑。

*   React 约束（如使用 React）：
    *   禁止在 render 中做副作用（请求、读写存储、订阅、埋点等）。
    *   useEffect 依赖必须准确，且必须清理副作用（定时器/订阅）。
    *   列表渲染必须使用稳定 key（禁止随机数；index 仅在静态且不会重排时允许）。

## 七、质量规范（强制）

*   禁止修改已有的全局主题、样式，避免影响其他页面

*   异常处理: 对用户提示友好，禁止直接抛出异常堆栈
    *  全局异常捕获：全局使用 window.onerror 捕获未处理异常
    *  异步操作必须捕获异常：所有 fetch / axios 等调用必须包裹 try-catch 或 .catch()
    *  用户操作需防重复提交：按钮点击后立即置灰/显示 loading，但提交异常时必须恢复按钮状态

*   边界处理与鲁棒性：
    *  前端需对用户输入的必要字段做基础校验
    *  避免 null/undefined 错误
    *  TypeScript中避免使用 any 类型

*   浏览器兼容:
    *  兼容主流浏览器及主流版本
    *  对不支持的 API 提供优雅降级

*   性能保障: 确保性能良好
    *  列表渲染必须加 key（唯一 ID，非 index）
    *  大列表（超过500条）使用虚拟滚动（如 react-window）
    *  对于大图片、视频等大资源，使用懒加载方式
    *  避免在 render 中创建新函数/对象
    *  批量修改 DOM：使用 DocumentFragment 或离线 DOM 操作
    *  避免触发布局抖动：读取样式后统一修改，而非读写交替
    *  接口必须设置超时（默认 10s）

*   安全保障：避免安全漏洞
    *   XSS攻击防御
